01 — how i think

I don't usually start with "what should I build?"

Most of what I work on starts from something personal — a problem I've lived through, or a workflow that quietly wastes everyone's time. I care less about how technically impressive something is, and more about whether it makes one specific part of someone's day less frustrating. I learn by breaking things and writing down what I notice, mostly on LinkedIn.

The first draft is usually just a question in the margin: why is this painful, and can software make it slightly less annoying?
  • campusUseful information exists, but it is usually scattered across portals, PDFs, emails, and screenshots.
  • queuesPeople can handle waiting better when the system is honest about what is happening.
  • securityNetworks work because of quiet trust assumptions. I like poking at those assumptions carefully.
  • student lifeMany student problems are not huge. They are small repeated frictions that slowly become exhausting.

Haajar (ഹാജർ)

chrome extension · personal project

"Students shouldn't have to make attendance decisions blindly."

Attendance portals showed a number, not a decision — so I built a privacy-first Chrome extension that reads attendance data straight from the university portal, maps it against the timetable, and tells students how many classes they can safely miss. Zero backend, zero external calls, zero data collection. It also syncs holidays and calendar events, supports batch timetable imports, and handles the messy edge cases: electives, inconsistent subject codes, internal exam windows.

Fog Scheduler Using Quantum-Inspired Principles

Sep 2025 — Apr 2026 · vehicular fog computing

"Why do scheduling algorithms break down under realistic vehicular workloads?"

Most scheduling papers test against neat, static assumptions — real vehicles don't move that way. I built a hybrid quantum-inspired genetic algorithm for task allocation across edge, fog, and cloud, then paired it with a SUMO–EdgeSimPy co-simulation so it had to handle actual mobility-driven workload shifts. Benchmarked against 10 baseline schedulers: up to 66% fewer SLA violations under dynamic load, and up to 93% lower energy use in static scenarios.

Python EdgeSimPy SUMO Streamlit Quantum-inspired optimisation

MediQ

May 2026 — present · real-time systems

"Hospital waiting shouldn't feel confusing and uncertain."

Waiting is worse when you don't know where you stand. MediQ is a serverless hospital queue system covering online appointments, walk-ins, live queue movement, and ETA prediction — built around actual clinic workflows (recovery after a refresh, skipped tokens, staff roles) rather than a tidy demo case. Patient identity and queue state are kept in separate Firestore collections, so the tracking link stays public without exposing who's who.

Firebase Firestore Real-time queues ETA prediction Security Rules

DNS Spoofing Detection & Alert System

CIAL internship · security

"Public wi-fi trusts a lot more than it should."

Built during my internship at Cochin International Airport, watching how much public wi-fi quietly trusts. This tool inspects DNS traffic at the packet level, checks suspicious responses against trusted resolvers, and flags transaction mismatches and TTL anomalies in real time — with a live dashboard, persistent forensic logs, and Telegram alerts so a detection actually turns into something investigable.
